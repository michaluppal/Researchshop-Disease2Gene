## Section 1 вАФ Run summary (verdict)

The pipeline **produced 3 genes** (ITPKC, CASP3, FCGR2A) from a Kawasaki Disease review
paper where PubTator had identified the same 3. All deterministic-lexicon candidates
without LLM or PubTator backing were correctly dropped by the corroboration gate вАФ 26 of
29 total candidates. That's the gate working exactly as designed, but the consequence on
a review paper with broad inflammation content is visible: plausible genes (NLRP3, TLR4,
TLR2, TNF, MYD88, CD14, HMGB1вА¶) were filtered out because PubTator didn't annotate them
and the LLM never got to score them.

**The LLM never got to score them** because **Gemini's free-tier daily quota was
exhausted at the very first API call of the run** (429 `RESOURCE_EXHAUSTED` on the
abstract pass). Every subsequent LLM call through the full-text passes, figure analysis,
and detail extraction also 429'd after exponential-backoff retries. The pipeline
gracefully degraded to:

- **Abstract pass** вАФ retries eventually succeeded (got 3 associations the same PubTator
  had already found)
- **Full-text passes (greedy + recall)** вАФ 3 retries each, all 429
- **Figure analysis** вАФ 3 retries √Ч 2 figures, all 429
- **Detail extraction** вАФ 3 retries, all 429. Fell back to `association_only_fallback_no_rows`
  then to auto-snippet keyword search for evidence backfill.

The final CSV has:

- вЬЕ Gene identity (HGNC-validated, confidence 1.0)
- вЬЕ `Key Finding` (auto-snippet вАФ same sentence excerpt for all 3 genes)
- вЭМ `Disease Association` (empty for all 3)
- вЪ†пЄП `Confidence Note` вАФ "Citation text not found in paper" (SequenceMatcher threshold
  miss; not actually absence of evidence вАФ see F10a and new F11 below)

**Trustworthy:** gene identity, existence check, metadata.
**Untrustworthy:** anything implying the LLM synthesized understanding of those genes.
**Missing:** per-gene evidence rows; disease linkage.

The run exercised the pipeline honestly вАФ every degradation path that triggered was
supposed to trigger under this condition. Nothing crashed, nothing silently corrupted,
the drop-debug artifact is complete, all 3 output artefacts wrote successfully.

## Section 4 вАФ Findings & fix priorities

### Severity ranked list

#### HIGH

**H1. Gemini quota exhaustion cascades to empty Disease Association column.**
When Gemini 429s, detail extraction falls back to `association_only_fallback_no_rows`,
which emits skeleton rows with all user-column values empty. The evidence-backfill step
then populates `Key Finding` via keyword search (auto-snippet) but does **not** attempt
to populate any other user column. The output CSV looks complete (`Key Finding` is
filled) but `Disease Association` is silently empty. There is no column or banner in the
output that signals "Gemini failed вАФ Key Finding is keyword fallback, Disease Association
is missing."

- Evidence: `final_enriched_results_c08ea1af.csv`, column `Disease Association` empty for
  all 3 rows; `Confidence Note = "Citation text not found in paper"`; `drop_debug_58d75316.json::paper_debug[0]`
  shows `detail_extraction_status = "fallback_after_retries"` with full 429 quota error
  preserved in `detail_extraction_error`.
- Cross-ref: вЖТ **new F11** in docs/audit/final-audit.md.
- Suggested fix: (a) add a `extraction_mode` column that records per-row whether
  content came from LLM or keyword fallback; (b) when fallback triggers, emit a
  top-level warning that reaches the UI.

**H2. Corroboration gate drops plausible Kawasaki genes under LLM failure.**
26 deterministic-lexicon-only candidates dropped. On this paper they include NLRP3,
TLR4, TLR2, TNF, MYD88, CD14, HMGB1 вАФ all plausibly relevant to Kawasaki Disease
immunopathogenesis. Under normal operation the LLM text passes would have corroborated
some of these (promoting them to multi-source вЖТ pass). Under Gemini quota failure, the
corroboration gate has no LLM input and the deterministic-only floor becomes the
sole result.

- Evidence: `drop_debug_58d75316.json` has 26 entries with `reason = "deterministic_uncorroborated"`.
- Cross-ref: F9 in docs/audit/final-audit.md (already documented).
- Suggested fix: when the LLM text-pass specifically failed (not just
  "didn't fire"), loosen the corroboration threshold to allow single-source
  deterministic hits to pass with a clear `deterministic_only` marker column and a
  confidence downgrade. Today the failure is invisible to the gate.

#### MEDIUM

**M1. All survivor rows share identical `Key Finding` excerpt.**
The keyword fallback in `_backfill_sparse_row_evidence` found the same sentence for all
3 genes because the paper has one summary sentence listing all three: *"Furthermore,
genetic polymorphisms associated with KD, such as ITPKC, CASP3, and FCGR2A, contribute
to immune activation by promoting inflammasome activation, pyroptosis and antibody
dependent enhancement (ADE), thereby intensifyingвА¶"* The CSV presents this as 3 rows of
gene-specific evidence; it's actually 1 sentence √Ч 3 genes.

- Evidence: `final_enriched_results_c08ea1af.csv`, rows 1-3 all start with the same
  `Key Finding`.
- Cross-ref: вЖТ **new F12** in docs/audit/final-audit.md. Related to F7 (per-gene detail extraction).
- Suggested fix: when the same snippet is backfilled for multiple rows, either (a)
  merge them into one row with aggregated variant names (which would collapse 3 gene
  rows into one "ITPKC; CASP3; FCGR2A" row вАФ unusual but honest), or (b) do a
  per-gene keyword search that requires the gene name itself to be in the sentence,
  not just any gene from the set.

**M2. Citation validation rejects the auto-snippet as "not found".**
After evidence backfill puts a paper-text snippet into `Key Finding Citation`, the
subsequent `_add_citation_validation_metadata` runs `_citation_exists_in_paper` which
uses SequenceMatcher вЙ• 0.85 and fails the match (likely due to whitespace/encoding
drift between the sliced snippet and the source). All 3 rows get
`citation_valid=False` and the final `Confidence Note` reads "Citation text not found in
paper" вАФ which is actively misleading because the snippet *is* from the paper, the
validator just couldn't match it back.

- Evidence: `final_enriched_results_c08ea1af.csv` column `Confidence Note`.
- Cross-ref: F10a in docs/audit/final-audit.md (existing). Plus new F11.
- Suggested fix: when evidence backfill puts a snippet into a citation cell, mark it
  as pre-validated (since we just copied it from the paper) and skip re-validation, OR
  use exact-substring check instead of fuzzy SequenceMatcher for this specific code
  path.

**M3. Stage-to-function time-window alignment is lossy.**
The stage markers captured via `pipeline_tracer.capture()` and the function-level
`fn_call`/`fn_return` events share the same live file but aren't linked by anything
other than timestamp. A stage marker for `detail_extraction` at T has function calls
(like `extract_gene_info`) that fire shortly after at T+50ms вАФ but the viewer groups
them by time windows rather than explicit parent-child. On this run, stages
`hgnc_validation`, `low_confidence_gate`, `detail_extraction`, `row_merge` reported 0
events even though their underlying function calls WERE captured вАФ they just happened
to fall into neighbouring stage windows.

- Evidence: agent coverage analysis reported "empty stages" that aren't actually empty;
  the function calls are in the event stream.
- Suggested fix: attach a `stage_id` field to every `fn_call`/`fn_return` event so the
  viewer can explicitly group function events under their parent stage. Add a stack
  variable in the tracer that tracks "currently active stage", pushed/popped by
  `capture()`.

#### LOW

**L1. 7 orchestrator helpers are observed name-only but carry useful values.**
`_finalize_paper_result`, `_get_citation_record`, `_write_split_output`,
`_run_pipeline_worker`, and the 5 nested getters in `_finalize_paper_result`
(`get_gene_source`, `get_ncbi_id`, `get_full_name`, `get_aliases`, `get_chromosome`) are
all called per row with meaningful inputs/outputs but appear in the trace name-only.

- Suggested fix: add to `_FN_TRACER_VALUE_CAPTURE` in `pipeline_tracer.py`. Covered in
  Section 3's рЯУЛ column.

**L2. `_collect_debug_artifact` is perhaps the single highest-value function not on
the watchlist.** It returns the complete `candidate_meta` state at end-of-run вАФ every
candidate's full provenance, drop reason, normalisation applied, raw labels. Clicking
this row in the viewer would show the whole lifecycle in one place.

- Suggested fix: add to the watchlist.

**L3. NCBI Gene enrichment is silently rate-limited too.** Only 1/3 survivor genes got
full NCBI Gene metadata (chromosome, aliases, full name) because the enrichment calls
also hit rate limits (429 on eutils). The missing columns in the final CSV are rendered
empty with no indication.

- Evidence: log line *"NCBI enrichment: Added metadata for 1/3 genes"*.
- Suggested fix: expose rate-limit failures in a `ncbi_enrichment_status` column so the
  user knows "row has no chromosome because the API 429'd", not "row was never
  enriched".
